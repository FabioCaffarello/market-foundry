# First Codegen-First Family: Generated vs Manual Boundaries

## Family: `bollinger` (Bollinger Bands — Signal Layer)

This document defines the explicit boundary between what is generated by the codegen pipeline and what remains manually authored for the first codegen-first family.

## Generated (Codegen-Governed)

These artifacts are **derived deterministically from the YAML spec** and must not be manually edited inside codegen markers.

### 1. Consumer Spec Function

**File:** `internal/adapters/nats/natssignal/registry.go`
**Markers:** `codegen:begin/end consumer_spec family=bollinger`

```go
func WriterBollingerSignalConsumer() natskit.ConsumerSpec {
    return natskit.ConsumerSpec{
        Durable: "writer-signal-bollinger",
        Event: natskit.EventSpec{
            Subject: "signal.events.bollinger.generated.>",
            Type:    "signal.events.v1.bollinger_generated",
            Stream:  natskit.StreamSpec{Name: "SIGNAL_EVENTS"},
        },
        AckWait:    30 * time.Second,
        MaxDeliver: 5,
    }
}
```

**What the spec controls:** durable name, NATS subject, event type, stream name.
**What is fixed by template:** ack wait (30s), max deliver (5).

### 2. Pipeline Entry

**File:** `cmd/writer/pipeline.go`
**Markers:** `codegen:begin/end pipeline_entry family=bollinger`

```go
{
    family:        "bollinger",
    consumerName:  "writer-signal-bollinger-consumer",
    inserterName:  "writer-signal-bollinger-inserter",
    table:         "signals",
    insertSQL:     "INSERT INTO signals (...columns...)",
    consumerSpec:  natssignal.WriterBollingerSignalConsumer(),
    isEnabled:     func(p settings.PipelineConfig) bool { return p.IsSignalFamilyEnabled("bollinger") },
    startConsumer: writerpipeline.NewSignalStarter(reg.signal),
}
```

**What the spec controls:** family name, table, columns, consumer/inserter names.
**What is derived from layer:** IsEnabled method, starter function, package alias.

### 3. Golden Snapshots

- `codegen/golden-snapshots/bollinger/consumer_spec.go.golden`
- `codegen/golden-snapshots/bollinger/pipeline_entry.go.golden`

These are the **source of truth** for what the generated output should look like. CI validates that production code inside markers matches these exactly.

### 4. Manifest Entry

`codegen/integrated.yaml` — two entries mapping bollinger artifacts to their targets.

## Manual (Human-Authored)

These artifacts require human judgment, domain knowledge, or architectural decisions that the spec does not encode.

### 1. Domain Logic: Bollinger Sampler

**File:** `internal/application/signal/bollinger_sampler.go`

**Why manual:** The Bollinger Bands algorithm (SMA, standard deviation, %B computation) is behavioral logic. The codegen spec captures structural wiring (NATS subjects, table names, consumer config) but not indicator algorithms. Generating domain logic from a spec would require encoding mathematical formulas in YAML, which is outside the codegen scope.

**Specific decisions:**
- Period = 20 (standard Bollinger default)
- K = 2.0 (standard deviation multiplier)
- Output value = %B (percent bandwidth)
- Rolling window implementation
- Metadata fields: sma, upper, lower, bandwidth, period, k

### 2. Unit Tests

**File:** `internal/application/signal/bollinger_sampler_test.go`

**Why manual:** Tests encode domain expectations (warm-up behavior, constant prices → %B=0.5, extreme prices → %B>1). These are inherently tied to the algorithm, not the wiring.

### 3. Registry Entries (Non-Writer)

**File:** `internal/adapters/nats/natssignal/registry.go` (outside codegen markers)

- `BollingerGenerated` EventSpec — manual because publish subjects don't use wildcard suffix
- `BollingerLatest` ControlSpec — manual because query/reply subjects follow a different pattern
- `LatestSpecByType("bollinger")` switch case
- `StoreBollingerSignalConsumer()` — manual because store consumers have different durable names and ownership

**Why manual:** The codegen system only governs **writer** consumer specs. Store consumers, event specs for publishing, and query specs serve different roles with different naming patterns and are maintained separately.

### 4. Config Registration

**File:** `internal/shared/settings/schema.go`

- `knownSignalFamilies["bollinger"] = true`
- `signalDependsOnEvidence["bollinger"] = []string{"candle"}`

**Why manual:** Config validation rules encode cross-layer dependency semantics. The spec doesn't capture what evidence families a signal depends on — that's a domain decision.

## Boundary Summary

| Artifact | Ownership | Source of Truth |
|---|---|---|
| Writer consumer spec function | **Generated** | YAML spec → template → golden |
| Pipeline entry struct | **Generated** | YAML spec → template → golden |
| Golden snapshots | **Generated** | codegen generate output |
| Manifest entry | **Generated** | codegen/integrated.yaml |
| Bollinger sampler (algorithm) | **Manual** | bollinger_sampler.go |
| Sampler unit tests | **Manual** | bollinger_sampler_test.go |
| Registry EventSpec/ControlSpec | **Manual** | registry.go (outside markers) |
| Store consumer spec | **Manual** | registry.go (outside markers) |
| Config known families | **Manual** | schema.go |
| Config dependency rules | **Manual** | schema.go |

## Boundary Invariant

> The codegen pipeline governs **structural wiring** (how events flow from NATS to ClickHouse).
> Human engineers govern **behavioral logic** (what the signal means and how it is computed).
> These two concerns never overlap within a single artifact.
