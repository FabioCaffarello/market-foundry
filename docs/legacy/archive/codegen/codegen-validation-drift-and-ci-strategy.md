# Codegen — Validation, Drift Detection, and CI Strategy

## Purpose

This document defines how generated code is validated against hand-crafted families, how drift between templates and generated output is detected, and how codegen enters the CI pipeline without becoming a black-box build step.

## Validation Strategy

### Golden Test Equivalence

The primary validation mechanism is **golden test equivalence**: the codegen tool, when given a family spec that describes an existing hand-crafted family, must produce output that is structurally equivalent to the hand-crafted code.

#### What "Structurally Equivalent" Means

Equivalence is tested at the **semantic structure** level, not byte-for-byte identity. Acceptable differences:

| Dimension | Allowed Difference |
|-----------|-------------------|
| Import ordering | Go-standard reordering |
| Whitespace/formatting | `gofmt`-normalized differences |
| Comment phrasing | Template comments vs hand-written |
| Variable naming | Must match; deviations are failures |
| Function signatures | Must match exactly |
| Column ordering in SQL | Must match DDL order |
| Test case ordering | Allowed to differ |
| Test case coverage | Generated must be ≥ hand-crafted |

Unacceptable differences:
- Missing fields in generated structs
- Different SQL query shape
- Different validation rules
- Different error handling paths
- Extra/missing imports that change behavior

#### Golden Spec Files

For each of the 6 existing families, a golden spec file will be created:

```
codegen/golden/
├── candle.yaml          # spec for L1 Evidence / Candle
├── rsi.yaml             # spec for L2 Signal / RSI
├── rsi_oversold.yaml    # spec for L3 Decision / RSI Oversold
├── mean_reversion.yaml  # spec for L4 Strategy / Mean Reversion Entry
├── position_exposure.yaml # spec for L5 Risk / Position Exposure
└── paper_order.yaml     # spec for L6 Execution / Paper Order
```

#### Golden Test Procedure

```
1. Read golden spec file (e.g., codegen/golden/rsi.yaml)
2. Run codegen tool → produces output files in temp directory
3. Compare each output file against hand-crafted file:
   a. Both through gofmt normalization
   b. Structural diff (AST-level for Go files, normalized for SQL/YAML)
   c. Report diffs that exceed allowed variance
4. Test passes if all output files are within equivalence bounds
5. Test fails if any structural deviation is detected
```

#### When Golden Tests Run

- On every modification to templates in `codegen/templates/`
- On every modification to golden spec files in `codegen/golden/`
- On PR that touches any codegen infrastructure
- Not on every commit (expensive; reserved for codegen-related changes)

### Compilation Gate

Every generated Go file must:
1. Compile without errors (`go build ./...`)
2. Pass `go vet ./...`
3. Not introduce new `golint` warnings

This is the minimum bar. It catches template syntax errors and type mismatches but does not validate behavioral correctness.

### Unit Test Gate

Generated test files (mapper tests, reader tests, use case tests, handler tests) must:
1. Pass when run against their generated implementation
2. Cover the same cases as the golden reference tests
3. Not depend on any codegen runtime library

### Integration Validation

For the first generated family (EMA Crossover, S193–S194):
1. Writer pipeline entry activates and connects to NATS
2. Events flow through consumer → mapper → inserter → ClickHouse
3. Existing `SignalReader` returns EMA crossover results via `type=ema_crossover`
4. Smoke test phase validates HTTP endpoint end-to-end
5. Health trackers report metrics for the new pipeline

## Drift Detection

### What Is Drift

Drift occurs when:
1. A template is modified but existing generated files are not regenerated.
2. A generated file is manually edited, breaking the spec → output contract.
3. A dependency of generated code (domain types, shared helpers) changes in a way that invalidates the generated output.

### Drift Detection Mechanisms

#### 1. Header Comment Check

Every generated file contains a header:
```go
// Code generated by mf-codegen from codegen/families/ema_crossover.yaml. DO NOT EDIT.
// Template version: v1.0.0
// Generated at: 2026-03-20T14:00:00Z
```

A CI check verifies:
- All files with the codegen header have a matching spec file.
- The template version in the header matches the current template version.
- If versions disagree, the file is flagged as potentially drifted.

#### 2. Regeneration Comparison

A CI job can regenerate all family specs and compare output against committed files:

```
for each codegen/families/*.yaml:
  regenerate → temp/
  diff committed/ temp/
  if diff: flag as drifted
```

This is the authoritative drift check. It catches:
- Template changes not propagated
- Manual edits to generated files
- Spec changes not regenerated

#### 3. Spec-to-Code Consistency

A lightweight check verifies that:
- Every spec file in `codegen/families/` has corresponding generated artifacts.
- Every generated artifact references a valid spec file.
- No orphaned generated files exist (spec deleted but file remains).

### Drift Response Protocol

| Drift Type | Severity | Response |
|------------|----------|----------|
| Template version mismatch | Warning | Regenerate and review diff |
| Manual edit to generated file | Error | Revert edit; apply fix to template or spec |
| Missing generated file | Error | Regenerate from spec |
| Orphaned generated file | Warning | Delete or create spec |
| Domain type change affecting mapper | Error | Update spec column definitions; regenerate |

## CI Strategy

### Principle: Transparent, Not Opaque

Codegen in CI must be **a verification step, not a build step**. Generated files are committed to the repository. CI verifies that committed files match what the codegen tool would produce. CI does not generate files during build.

Rationale:
1. Committed generated files are reviewable in PRs.
2. No build-time dependency on the codegen tool for regular development.
3. Drift detection is explicit, not implicit.
4. Developers can read and debug generated code without running codegen.

### CI Pipeline Integration

```
┌──────────────────────────────────────────────────────┐
│  Existing CI Pipeline                                 │
│                                                       │
│  go build → go vet → go test → (existing checks)     │
│                                                       │
├──────────────────────────────────────────────────────┤
│  Codegen Verification (new, additive)                 │
│                                                       │
│  1. codegen-lint:                                     │
│     - Validate all spec files (schema check)          │
│     - Verify header comments in generated files       │
│     - Check for orphaned/missing files                │
│                                                       │
│  2. codegen-golden (on template/spec changes):        │
│     - Regenerate golden specs → compare with hand-    │
│       crafted families                                │
│     - Fail if structural equivalence is violated      │
│                                                       │
│  3. codegen-drift (on any change):                    │
│     - Regenerate all family specs                     │
│     - Compare with committed generated files          │
│     - Fail if drift detected                          │
│                                                       │
│  4. codegen-compile:                                  │
│     - go build on generated files                     │
│     - go test on generated test files                 │
│     (This is already covered by existing go build/    │
│      test steps — no new step needed)                 │
│                                                       │
└──────────────────────────────────────────────────────┘
```

### CI Step Costs

| Step | Trigger | Duration (est.) | Blocking? |
|------|---------|----------------|-----------|
| codegen-lint | Every PR | <5s | Yes |
| codegen-golden | Template/spec changes | ~10s | Yes |
| codegen-drift | Every PR with generated files | ~10s | Yes |
| codegen-compile | Already covered by `go build` | 0 (no new cost) | N/A |

### What CI Does NOT Do

1. **CI does not run codegen to produce build artifacts.** Generated files are pre-committed.
2. **CI does not auto-fix drift.** Drift failures require human review and regeneration.
3. **CI does not validate ClickHouse integration.** That requires the compose stack (smoke test scope, not CI scope).
4. **CI does not enforce codegen for manual families.** The 6 existing families remain hand-crafted and are not subject to codegen checks.

## Validation Lifecycle Per Family

### New Within-Layer Family (Tier 1)

```
1. Author creates spec: codegen/families/ema_crossover.yaml
2. Run codegen tool locally → generates write-path artifacts
3. Developer reviews generated output
4. Commit spec + generated files
5. CI runs:
   a. codegen-lint: spec is valid
   b. codegen-drift: generated files match spec
   c. go build: compiles
   d. go test: unit tests pass
6. PR review: reviewer inspects generated code (it's committed, readable)
7. Merge → deploy
8. Smoke test: validates end-to-end data flow
```

### Template Modification

```
1. Developer modifies template in codegen/templates/
2. Run golden tests locally → verify equivalence with 6 families
3. Regenerate all family specs → update generated files
4. Commit template + regenerated files
5. CI runs:
   a. codegen-golden: equivalence maintained
   b. codegen-drift: all generated files up to date
   c. go build/test: no regressions
6. PR review: reviewer inspects template diff AND regenerated diffs
7. Merge
```

## Maturity Model

### Phase 1 (S193): Minimal Viable Codegen
- Tier 1 templates only (write-path)
- Golden tests against 6 families
- Manual CI verification (run codegen-lint locally)
- First generated family: EMA Crossover

### Phase 2 (post-S194): CI Integration
- codegen-lint and codegen-drift in CI pipeline
- Automated golden test on template changes
- Template versioning with header tracking

### Phase 3 (future): Tier 2 Templates
- Full read-path generation for new layers
- Expanded golden test coverage
- Codegen covers ~90% of family expansion cost
