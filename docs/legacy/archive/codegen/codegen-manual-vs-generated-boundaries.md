# Codegen Manual vs Generated Boundaries

> Companion to [codegen-specification-freeze.md](codegen-specification-freeze.md).
> Defines the exact boundary between what is generated by codegen and what remains human responsibility.

## Governing Principle

Codegen generates **only** artifacts that satisfy all three conditions:

1. **Repetitive** — implemented 3+ times with identical structure across families.
2. **Mechanical** — requires zero creative or architectural decisions.
3. **Spec-derivable** — every value in the output comes directly from the spec file, with no inference or context beyond the spec.

If any condition is not met, the artifact is human-owned.

## The Boundary Map

### Always Human-Owned

These artifacts are never generated, regardless of tier or family count. Each entry includes the reason this side of the boundary was chosen.

| Artifact | Location | Why Human-Owned |
|----------|----------|-----------------|
| **Domain event types** | `internal/domain/{layer}/` | Defines the domain model. Requires understanding of business semantics, field relationships, and type design. Architectural decision. |
| **NATS stream definitions** | NATS server config / adapter setup | Stream topology is infrastructure architecture. Affects durability, partitioning, and operational semantics. |
| **Writer core logic** | `cmd/writer/consumer.go`, `inserter.go`, `supervisor.go` | Shared infrastructure implementing the pipeline engine. Modified only for cross-cutting concerns. |
| **ClickHouse client** | `internal/adapters/clickhouse/client.go` | Connection management, pooling, error handling. Infrastructure concern. |
| **Health/observability framework** | `internal/shared/healthz/` | Cross-cutting infrastructure shared by all services. |
| **Gateway composition root** | `cmd/gateway/compose.go` | Service wiring requires understanding of dependency graph and startup order. |
| **HTTP server/router setup** | `cmd/gateway/run.go` | Framework-level concern with cross-cutting middleware. |
| **Shared helpers** | `parseFloat`, `marshalJSON`, `parseAnalyticalParams` | Used by multiple families; changes affect all consumers. Human review required. |
| **CI configuration** | `.github/workflows/` | Affects build, test, and deployment for entire repository. |
| **Migration DDL (Tier 1)** | `deploy/migrations/` | For within-layer families, the table already exists. Schema decisions are architectural. |
| **Template files** | `codegen/templates/` | Templates define the codegen's behavior. Meta-level human ownership. |
| **Spec files** | `codegen/families/` | Specs encode architectural decisions. Always human-authored. |

### Codegen-Owned (Tier 1)

These artifacts are generated when a valid spec file exists. Codegen owns the file content; humans own the spec and template that produce it.

| Artifact | Location | Why Generated |
|----------|----------|---------------|
| **Writer consumer spec** | `internal/adapters/nats/{domain}_registry.go` | Mechanical: subject + durable + stream → consumer config. Identical structure for every family. |
| **Writer pipeline entry** | `cmd/writer/pipeline.go` | Mechanical: family key + table + SQL + consumer + mapper → pipeline declaration. |
| **Writer mapper** (conditional) | `cmd/writer/mappers.go` | When event struct differs from existing mapper, column extraction is mechanical if columns are declared in spec. Only generated when `mapper: "generate"`. |
| **Writer mapper tests** (conditional) | `cmd/writer/mappers_test.go` | Test cases derived directly from declared columns. Happy path + edge cases are deterministic. |
| **Writer config entry** | `deploy/configs/writer.jsonc` | Appending a family name to a config array. Pure mechanical. |
| **Smoke test phase** | `scripts/smoke-analytical-e2e.sh` | Endpoint URL + required fields + filter tests. Follows identical pattern for every family. |

### Codegen-Owned (Tier 2 — Not Yet Authorized)

These artifacts will be generated when Tier 2 is activated. Listed here for completeness and boundary clarity.

| Artifact | Location | Why Generated (Future) |
|----------|----------|------------------------|
| Migration DDL | `deploy/migrations/` | For new-layer families, DDL is declared in the spec's `schema` section. |
| Reader adapter method | `internal/adapters/clickhouse/` | Already type-parameterized; new family requires mechanical wiring. |
| Reader adapter tests | Same | Deterministic test cases from column list. |
| Use case | `internal/application/analyticalclient/` | Delegates to reader, assembles reply. Identical pattern. |
| Use case tests | Same | Mock reader, verify delegation. Identical pattern. |
| Contracts (query/reply) | `internal/application/analyticalclient/contracts.go` | Struct with fields derived from spec. |
| Handler method | `internal/interfaces/http/handlers/analytical.go` | Parse params, call use case, format response. Identical pattern. |
| Handler tests | Same | Valid/invalid param combinations. Deterministic. |
| Route registration | `internal/interfaces/http/routes/analytical.go` | Single line wiring handler to path. |
| Gateway reader factory | `cmd/gateway/compose.go` (append section) | Factory construction from ClickHouse client. |
| Gateway wiring | `cmd/gateway/compose.go` (append section) | Dependency injection entry. |

## Decisions That Are Always Human

These decisions are never automated, inferred, or defaulted by codegen. Each must be explicitly made by a human before a family can be generated.

| Decision | Why Human | Where Expressed |
|----------|-----------|-----------------|
| **Which event types become analytical families** | Business and architectural judgment about what data is worth persisting and querying. | Decision to create a spec file. |
| **Domain type design** | Field names, types, semantics, relationships — core domain modeling. | `internal/domain/{layer}/` source code. |
| **Schema design for new layers** | Table structure, column types, partitioning, ordering — ClickHouse expertise required. | `deploy/migrations/` and spec `schema` section. |
| **Stream and subject naming** | NATS topology decisions affect routing, filtering, and operational semantics. | NATS config and spec `nats` section. |
| **Which filters an endpoint supports** | API design decision affecting consumer contracts. | Handler implementation (Tier 2) or existing handler. |
| **When to create new table vs reuse existing** | Storage architecture decision with operational implications. | Choice between `tier: 1` and `tier: 2`. |
| **Mapper field transformations** | Deciding which fields need `parseFloat`, `marshalJSON`, etc. requires understanding of type semantics. | Spec `domain.columns[].transform` field. |
| **Retry/backoff policy per family** | Operational decision based on event volume, criticality, and failure characteristics. | Infrastructure configuration, not per-family. |
| **Whether a family warrants codegen or manual implementation** | Meta-decision about complexity and edge cases. | Decision to create spec file (codegen) or implement manually. |
| **Template evolution** | Changing how codegen works is an architectural decision affecting all generated families. | Template file modifications with golden test validation. |

## Decisions Codegen Can Make

These are mechanical, deterministic operations with zero ambiguity:

| Decision | Why Mechanical | Input |
|----------|----------------|-------|
| Consumer spec construction | Mapping 3 spec fields to a Go struct literal. | `nats.subject`, `nats.durable`, `nats.stream` |
| Pipeline entry wiring | Mapping 5 spec fields to a function call. | `writer.*` fields |
| Mapper column ordering | Extracting fields in declared order. | `domain.columns[]` |
| Test case generation | Happy path + zero-value + empty-object cases for each column. | `domain.columns[]` |
| Config array entry | Appending a string to a JSON array. | `family.name`, `writer.config_array` |
| Smoke test phase | URL construction + field assertions from spec. | `family.*`, endpoint pattern |

## The Boundary Test

When evaluating whether an artifact or decision should cross the boundary:

```
1. Has this been implemented ≥3 times with identical structure?
   NO  → remains manual
   YES → continue

2. Does every value come directly from the spec, with no inference?
   NO  → remains manual
   YES → continue

3. Could a wrong version cause silent data corruption?
   YES → generate WITH golden test validation, OR remain manual
   NO  → continue

4. Does this define an API contract visible to external consumers?
   YES → generate but require human review before merge
   NO  → generate

5. All conditions met → candidate for generation
```

## Append-Only Integration Protocol

Where codegen must add entries to human-owned files, the following protocol applies:

### Marker Format

```go
// --- BEGIN CODEGEN MANAGED SECTION ---
// Do not manually edit between these markers.
// Managed by mf-codegen. Changes here will be overwritten on regeneration.

// ... generated entries ...

// --- END CODEGEN MANAGED SECTION ---
```

### Rules

1. **Markers are placed once, manually**, during initial codegen setup. Codegen never creates markers in files that don't already have them.
2. **Content between markers is fully codegen-owned.** Regeneration replaces all content between markers.
3. **Content outside markers is fully human-owned.** Codegen never reads, modifies, or moves content outside markers.
4. **One marker pair per integration point.** No nested or multiple marker sections in a single file.
5. **Marker sections are append-only in semantics.** Each regeneration produces the complete set of entries for all active specs, sorted deterministically.

### Integration Points Using Markers

| File | Section Purpose | Entry Format |
|------|-----------------|--------------|
| `cmd/writer/pipeline.go` | Pipeline entry declarations | `WriterPipeline{...}` struct literals |
| `cmd/writer/mappers.go` | Generated mapper functions | `func map<Family>Row(...)` |
| `cmd/writer/mappers_test.go` | Generated mapper test functions | `func TestMap<Family>Row(...)` |
| `deploy/configs/writer.jsonc` | Family array entries | String additions to config array |
| `scripts/smoke-analytical-e2e.sh` | Smoke test phases | Bash test sections |
| `internal/adapters/nats/{domain}_registry.go` | Consumer spec functions | `func <Family>ConsumerSpec()` |

## What Stays Manual by Strategic Choice

Some artifacts could theoretically be generated but are deliberately kept manual:

| Artifact | Could Generate? | Why Kept Manual |
|----------|:-:|----------------|
| Golden spec files | No | They describe existing code; reversing the relationship would lose the validation purpose. |
| Migration DDL (Tier 1) | Partially | Within-layer families reuse existing tables. The decision to reuse vs create is architectural. |
| Reader adapters (Tier 1) | Yes, trivially | Already type-parameterized and generic. No new code needed per within-layer family. Generation would be zero-value. |
| Existing 6 families | Yes | Proven in production, tested, stable. Retroactive regeneration adds risk with zero benefit. |
| `compose.go` wiring | Partially | Service composition involves startup ordering, error handling, and dependency management beyond mechanical wiring. |

## Boundary Violations and Response

| Violation | Detection | Response |
|-----------|-----------|----------|
| Manual edit to generated file | CI regeneration comparison | Revert edit; apply fix to template/spec; regenerate |
| Codegen modifies human-owned code outside markers | Code review; markers as enforcement | Fix codegen; never allow marker-less writes to human files |
| Codegen infers a value not in spec | Golden test (would produce different output than hand-crafted) | Fix template to require explicit spec field |
| Generated file missing header | CI header check | Fix template to include header |
| Spec contains undocumented field | Spec schema validation | Reject spec; add field to schema if justified |

## Summary

The boundary between manual and generated is not a gradient. It is a hard line:

- **Left of the line**: human decisions, architectural judgment, domain modeling, infrastructure design.
- **Right of the line**: mechanical, repetitive, spec-derivable artifacts with zero creative content.

Codegen operates exclusively on the right side. It never reaches across.
