# Venue-Integrated Execution Design

> Design document for crossing the action boundary — from paper execution to venue-integrated execution.
> Date: 2026-03-18 | Stage: S75
> Classification: DESIGN-ONLY — no implementation in this stage.

---

## 1. Purpose

This document designs the architecture for venue-integrated execution in Market Foundry. It addresses all 5 hard blockers identified by S74 and provides concrete contracts, transition rules, and patterns that subsequent implementation stages (S76–S80) will follow.

**What this document IS:**
- Port interface contracts
- Domain model extensions (lifecycle, fill tracking)
- Failure recovery patterns
- Kill switch mechanism
- Trace persistence strategy
- Staleness guard design

**What this document IS NOT:**
- Implementation code
- Venue API integration
- OMS, portfolio, or multi-venue routing
- Framework or generic abstraction

## 2. Architectural Decision: Venue Adapter Topology

### Decision

**Option C from S74/SR-1**: Separate `execute` binary.

```
derive                    NATS                      execute                   NATS                      store
──────                    ────                       ───────                   ────                      ─────
PaperOrderEvaluator       EXECUTION_EVENTS           VenueAdapterActor         EXECUTION_FILL_EVENTS     FillProjectionActor
  │ produces intent         │                          │ consume intent           │                        │ materialize fill
  ▼                         │                          │ place order              │                        ▼
ExecutionPublisher ──────►  │  ──────────────────────► │ receive fill             │  ◄─────────────────── FillPublisher
                            │                          │ publish fill event       │
                            │                          ▼                          │
                            │                        VenuePort (interface)        │
                            │                          │ submit order             │
                            │                          │ poll status              │
                            │                          ▼                          │
                            │                        Exchange API                 │
```

### Rationale

| Factor | Why Option C |
|--------|-------------|
| Derive stays pure | Derive produces intents. It never touches external APIs. |
| Store stays read-side | Store materializes projections. It never places orders. |
| Gateway stays stateless | Gateway translates HTTP. It never handles execution logic. |
| Failure isolation | Execute binary failure does not affect derive or store. |
| Kill switch scope | Kill switch targets execute binary only. Derive/store unaffected. |
| Independent scaling | Execute can have different resource/rate-limit profiles. |
| Audit boundary | Intent generation (derive) and order placement (execute) are separate audit events. |

### Binary Responsibilities

| Binary | Responsibility | Produces | Consumes |
|--------|---------------|----------|----------|
| derive | Generate execution intents | EXECUTION_EVENTS | Risk assessments (local actor) |
| execute | Place orders at venues | EXECUTION_FILL_EVENTS | EXECUTION_EVENTS |
| store | Materialize intents + fills | KV buckets | EXECUTION_EVENTS + EXECUTION_FILL_EVENTS |
| gateway | Query surface | HTTP responses | NATS request/reply to store |

### Alternatives Rejected

| Alternative | Rejection Rationale |
|-------------|---------------------|
| Venue adapter inside derive | Violates derive-is-pure principle. Introduces external failure modes into the analytical chain. |
| Venue adapter inside store | Violates store-is-read-side principle. Store materializes, it does not act. |
| Single binary (derive+execute) | Couples intent generation with order placement. Kill switch cannot target execution alone. |
| Sidecar pattern | Over-engineered for single-venue first step. Consider if multi-venue becomes reality. |

## 3. Execution Lifecycle State Machine

### Status Enum Extension

```go
type Status string

const (
    // Phase 1 (existing)
    StatusSubmitted Status = "submitted"   // Intent generated by derive

    // Phase 2 (venue integration)
    StatusSent      Status = "sent"        // Order sent to venue adapter
    StatusAccepted  Status = "accepted"    // Venue acknowledged receipt
    StatusFilled    Status = "filled"      // Order fully filled
    StatusPartial   Status = "partial"     // Order partially filled
    StatusRejected  Status = "rejected"    // Venue rejected order
    StatusCancelled Status = "cancelled"   // Order cancelled (by system or user)
    StatusExpired   Status = "expired"     // Order expired at venue
)
```

### Transition Rules

```
                    ┌──────────────┐
                    │  submitted   │  (derive produces intent)
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │     sent     │  (execute binary sends to venue)
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
       ┌──────▼───┐  ┌────▼─────┐  ┌───▼──────┐
       │ accepted  │  │ rejected │  │ expired  │
       └──────┬────┘  └──────────┘  └──────────┘
              │
         ┌────┼─────────┐
         │              │
    ┌────▼───┐    ┌─────▼─────┐
    │ filled │    │  partial  │──── (may transition to filled, cancelled, or expired)
    └────────┘    └───────────┘
                        │
              ┌─────────┼──────────┐
              │                    │
         ┌────▼───┐          ┌─────▼─────┐
         │ filled │          │ cancelled │
         └────────┘          └───────────┘
```

### Valid Transitions Table

| From | To | Trigger |
|------|----|---------|
| submitted | sent | Execute binary consumes intent |
| sent | accepted | Venue acknowledges order |
| sent | rejected | Venue rejects order |
| sent | expired | Venue timeout / TTL |
| accepted | filled | Full fill confirmation |
| accepted | partial | Partial fill confirmation |
| accepted | cancelled | Cancel request accepted |
| accepted | expired | Order TTL expired |
| partial | filled | Remaining quantity filled |
| partial | cancelled | Cancel remaining |
| partial | expired | Remaining expired |

### Invalid Transitions (Must Be Rejected)

- Any transition FROM a terminal state (filled, rejected, cancelled, expired)
- submitted → filled (skips sent+accepted)
- sent → partial (skips accepted)
- sent → filled (skips accepted)
- Any backward transition (accepted → sent, filled → accepted, etc.)

### Transition Validation Contract

```go
// ValidTransition reports whether transitioning from `from` to `to` is allowed.
func ValidTransition(from, to Status) bool {
    allowed, ok := transitions[from]
    if !ok {
        return false
    }
    for _, s := range allowed {
        if s == to {
            return true
        }
    }
    return false
}

// IsTerminal reports whether the status is a terminal (final) state.
func IsTerminal(s Status) bool {
    return s == StatusFilled || s == StatusRejected || s == StatusCancelled || s == StatusExpired
}
```

### Phase 1 Compatibility

Paper execution continues to use `StatusSubmitted` only. The lifecycle extension is additive — `ValidStatus()` accepts all values. Paper order evaluators never produce statuses beyond `submitted`. The execute binary is the sole producer of status transitions.

## 4. Fill Tracking Model

### Design Decision: Separate Entity

Fill data is modeled as a **separate entity** (`ExecutionFill`), not embedded in `ExecutionIntent`.

**Rationale:**
- Intents are produced by derive. Fills are produced by the execute binary. Different ownership.
- An intent may have 0 or N fills (partial fills produce multiple fill events).
- Fill data has different lifecycle than intent data.
- Separate projection: intent projection is latest-only; fill projection is append-only (history).

### ExecutionFill Entity

```go
type ExecutionFill struct {
    // Identity
    FillID       string    `json:"fill_id"`        // Unique fill identifier (UUID)
    VenueOrderID string    `json:"venue_order_id"`  // Venue's order reference
    IntentKey    string    `json:"intent_key"`      // Links to ExecutionIntent partition key

    // Fill data
    Source       string    `json:"source"`
    Symbol       string    `json:"symbol"`
    Timeframe    int       `json:"timeframe"`
    Side         Side      `json:"side"`
    FilledQty    string    `json:"filled_qty"`      // Quantity filled in this event
    FillPrice    string    `json:"fill_price"`      // Price at which fill occurred
    CumulativeQty string  `json:"cumulative_qty"`  // Total filled so far
    RemainingQty string   `json:"remaining_qty"`   // Remaining unfilled

    // Status
    OrderStatus  Status    `json:"order_status"`    // Current order status after this fill
    FillType     FillType  `json:"fill_type"`       // full, partial

    // Audit
    VenueTimestamp time.Time `json:"venue_timestamp"` // Venue-reported fill time
    Timestamp      time.Time `json:"timestamp"`       // System receive time

    // Trace
    CorrelationID string `json:"correlation_id"`
    CausationID   string `json:"causation_id"`
}

type FillType string

const (
    FillTypeFull    FillType = "full"
    FillTypePartial FillType = "partial"
)
```

### Fill Validation Rules

- `FillID` MUST be non-empty
- `VenueOrderID` MUST be non-empty
- `IntentKey` MUST be non-empty and match an existing intent partition key
- `Symbol` MUST be non-empty
- `FilledQty` MUST be non-empty and positive
- `FillPrice` MUST be non-empty and positive
- `CumulativeQty` MUST be >= `FilledQty`
- `OrderStatus` MUST be valid and represent a valid transition
- `VenueTimestamp` MUST be non-zero
- `CorrelationID` MUST be non-empty (links to originating intent's chain)

### Fill Event

```go
const EventExecutionFilled events.Name = "execution_filled"

type ExecutionFilledEvent struct {
    Metadata events.Metadata `json:"metadata"`
    Fill     ExecutionFill   `json:"fill"`
}
```

| Property | Value |
|----------|-------|
| Stream | `EXECUTION_FILL_EVENTS` |
| Subjects | `execution.fill.>` |
| Subject pattern | `execution.fill.{type}.{source}.{symbol}.{timeframe}` |
| Storage | FileStorage |
| MaxAge | 30 days (fills have longer audit retention) |
| MaxBytes | 4 GB |

### Fill Projection

| Bucket | Purpose | Key format |
|--------|---------|-----------|
| `EXECUTION_FILL_LATEST` | Latest fill per order | `{venue_order_id}` |

History bucket is deferred. First step uses latest-only, consistent with all other domains.

## 5. Venue Adapter Port Interface

### Port Definition

```go
// VenuePort is the port interface for order placement and status polling.
// Implemented by venue-specific adapters (e.g., Binance, paper simulator).
// Consumed by the execute binary's VenueAdapterActor.
//
// Location: internal/application/ports/venue.go
type VenuePort interface {
    // SubmitOrder sends an execution intent to the venue.
    // Returns a VenueOrderReceipt on success, or a problem on failure.
    // This is a synchronous call — it blocks until the venue responds or times out.
    SubmitOrder(ctx context.Context, req VenueOrderRequest) (VenueOrderReceipt, *problem.Problem)

    // GetOrderStatus polls the venue for the current status of an order.
    // Returns the current status, or a problem if the order is unknown or unreachable.
    GetOrderStatus(ctx context.Context, venueOrderID string) (VenueOrderStatus, *problem.Problem)

    // CancelOrder requests cancellation of an active order.
    // Returns the cancellation result, or a problem if cancellation failed.
    CancelOrder(ctx context.Context, venueOrderID string) (VenueCancelResult, *problem.Problem)
}
```

### Request/Response Types

```go
// VenueOrderRequest is the input to SubmitOrder.
type VenueOrderRequest struct {
    Symbol    string
    Side      execution.Side
    Quantity  string
    OrderType string            // "market" for first step; "limit" deferred
    Metadata  map[string]string // Trace context passthrough
}

// VenueOrderReceipt is the synchronous response from venue after submission.
type VenueOrderReceipt struct {
    VenueOrderID string          // Venue-assigned order identifier
    Status       execution.Status // Initial status (typically "accepted" or "rejected")
    Timestamp    time.Time        // Venue-reported timestamp
    Message      string           // Venue message (e.g., rejection reason)
}

// VenueOrderStatus is the response from GetOrderStatus.
type VenueOrderStatus struct {
    VenueOrderID  string
    Status        execution.Status
    FilledQty     string
    AverageFillPrice string
    RemainingQty  string
    LastUpdated   time.Time
}

// VenueCancelResult is the response from CancelOrder.
type VenueCancelResult struct {
    VenueOrderID string
    Cancelled    bool
    Message      string
    Timestamp    time.Time
}
```

### Adapter Implementations (Future)

| Adapter | Binary | Purpose |
|---------|--------|---------|
| `PaperVenueAdapter` | execute | Simulates venue locally (first implementation) |
| `BinanceFuturesAdapter` | execute | Real Binance Futures API integration |

**First implementation MUST be PaperVenueAdapter** — a local simulator that accepts all orders with configurable fill delay. This proves the execute binary topology without touching any real exchange.

### Port Invariants

| ID | Invariant |
|----|-----------|
| VP-1 | VenuePort implementations MUST be injected, never hard-coded. |
| VP-2 | VenuePort calls MUST have configurable timeouts. Default: 10 seconds. |
| VP-3 | VenuePort implementations MUST be idempotent for SubmitOrder (same intent = same result). |
| VP-4 | VenuePort implementations MUST NOT cache state. The execute binary owns state. |
| VP-5 | VenuePort implementations MUST return structured errors via problem.Problem. |
| VP-6 | VenuePort MUST NOT import from derive, store, or gateway packages. |
| VP-7 | All VenuePort calls MUST carry trace context (correlation_id, causation_id) in Metadata. |

## 6. Failure Recovery Pattern

### Current Failures (from S74 HB-3)

| Failure | Current Behavior | Impact |
|---------|-----------------|--------|
| NATS publish failure | Log + drop | Event permanently lost |
| KV write failure | Log + ACK | Message acknowledged despite data loss |
| Venue API timeout | N/A (no venue) | Future: order state unknown |
| Venue API error | N/A | Future: order may or may not have been placed |

### Pattern 1: Publish Retry with Backoff

**Scope**: All JetStream publish operations (derive publishers, execute fill publisher).

```
Publish attempt
  │
  ├── Success → done
  │
  ├── Transient error (timeout, connection reset) →
  │     Retry with exponential backoff
  │     Max retries: 3
  │     Base delay: 100ms
  │     Max delay: 2s
  │     Jitter: ±50ms
  │     │
  │     ├── Retry succeeds → done
  │     └── All retries exhausted → circuit breaker
  │
  └── Permanent error (invalid subject, auth failure) →
        Log + drop (no retry)
        Increment error counter
```

**Circuit Breaker**:
- After 3 consecutive publish failures on the same publisher: enter OPEN state
- OPEN state: reject all publishes immediately for 30 seconds
- After 30 seconds: attempt one publish (HALF-OPEN)
- Success: return to CLOSED
- Failure: stay OPEN for another 30 seconds
- Circuit breaker state changes MUST be logged at ERROR level with full context

**Implementation location**: Shared utility in `internal/adapters/nats/publish_retry.go`. All publishers wrap their `js.Publish` call with this utility.

### Pattern 2: Projection NAK on Write Failure

**Scope**: All projection actors (store binary).

**Current behavior** (BROKEN):
```
Consumer delivers message → Projection attempts KV write → KV write fails → Log error → ACK message
```

**Correct behavior**:
```
Consumer delivers message → Projection attempts KV write → KV write fails → NAK message → Redelivery
```

**Rules**:
- On KV write failure: call `msg.Nak()` (not `msg.Ack()`)
- NAK triggers redelivery up to MaxDeliver (5 attempts)
- After MaxDeliver exhaustion: message is dead-lettered by JetStream
- Dead-letter visibility: log at ERROR with all trace context + increment `deadLettered` counter in stats

**Stats extension** (all projection actors):
```go
type projectionStats struct {
    // existing
    received        atomic.Int64
    materialized    atomic.Int64
    skippedStale    atomic.Int64
    skippedDedup    atomic.Int64
    skippedNonFinal atomic.Int64
    rejected        atomic.Int64
    errors          atomic.Int64
    // new
    nakRetried      atomic.Int64  // NAK'd for redelivery
    deadLettered    atomic.Int64  // Exhausted MaxDeliver
}
```

**Stats invariant update**: `received == materialized + skippedStale + skippedDedup + skippedNonFinal + rejected + errors + nakRetried + deadLettered`

### Pattern 3: Execute Binary — Venue Failure Handling

**Scope**: Execute binary's VenueAdapterActor.

```
Consume ExecutionIntent from EXECUTION_EVENTS
  │
  ├── Call VenuePort.SubmitOrder()
  │     │
  │     ├── Success (VenueOrderReceipt) →
  │     │     Publish StatusSent event
  │     │     Publish StatusAccepted event (if receipt says accepted)
  │     │     ACK consumer message
  │     │
  │     ├── Timeout →
  │     │     DO NOT ACK — NAK for redelivery
  │     │     Log with full trace context
  │     │     On redelivery: check GetOrderStatus() first (idempotency guard)
  │     │
  │     ├── Rejected →
  │     │     Publish StatusRejected event
  │     │     ACK consumer message
  │     │
  │     └── Unknown error →
  │           DO NOT ACK — NAK for redelivery
  │           Log with full trace context
  │           After MaxDeliver: dead-letter
  │
  └── Idempotency guard on redelivery:
        Before calling SubmitOrder, check if VenueOrderID exists for this intent
        If exists → skip submission, poll status instead
        If not → submit as normal
```

### Pattern 4: Dead Letter Queue Visibility

**Design**: No separate DLQ stream in first step. Instead:

- JetStream MaxDeliver exhaustion is the implicit DLQ
- All exhaustion events logged at ERROR with:
  - correlation_id, causation_id
  - original subject
  - consumer durable name
  - delivery count
  - last error message
- A `deadLettered` stats counter per projection actor
- Query via health endpoint: `GET /healthz` includes dead-letter counts

**Future (S81+)**: Explicit DLQ stream with replay capability.

## 7. Staleness Guard Design

### Purpose

Prevent execution on risk assessments that are too old to be trustworthy.

### Mechanism

```go
// StalenessGuard rejects risk assessments older than the configured threshold.
// Location: internal/application/execution/staleness_guard.go
type StalenessGuard struct {
    MaxAge time.Duration
}

// Check returns nil if the assessment is fresh enough, or a problem if stale.
func (g StalenessGuard) Check(assessmentTime time.Time, now time.Time) *problem.Problem {
    age := now.Sub(assessmentTime)
    if age > g.MaxAge {
        return problem.New(problem.FailedPrecondition,
            fmt.Sprintf("risk assessment is stale: age=%s, max=%s", age, g.MaxAge))
    }
    return nil
}
```

### Configuration

```jsonc
// derive.jsonc
{
  "pipeline": {
    "execution_families": ["paper_order"],
    "execution_staleness_max_seconds": 120  // 2× default timeframe (60s)
  }
}
```

**Default**: 2× the timeframe. For a 60-second timeframe, max staleness = 120 seconds.

### Integration Point

Staleness check happens **inside the execution evaluator actor**, before calling the pure evaluator function:

```
PaperOrderEvaluatorActor.onRiskAssessed()
  │
  ├── staleness guard: reject if assessment too old
  │     └── Log stale rejection with trace context
  │
  ├── call PaperOrderEvaluator.Evaluate() (pure function, unchanged)
  │
  └── send publishExecutionMessage
```

The pure evaluator function (`PaperOrderEvaluator.Evaluate()`) is NOT modified. Staleness is an actor-level concern, not a domain-level concern.

### Stats

New counter in execution evaluator actor stats: `skippedStale`. Incremented on every stale rejection.

## 8. Kill Switch Mechanism

### Requirements (from S74 HB-5)

- Halt execution within 1 event cycle (< 1 second)
- No binary restart required
- Auditable (logged with timestamp)
- Recoverable (resume without restart)
- Scoped: halt execution only, not derive/store/gateway

### Design: NATS KV Signal

```
configctl                       NATS KV                          execute / derive
─────────                       ───────                          ────────────────
Set "execution.halted" = true   EXECUTION_CONTROL bucket         Watch key changes
                                  │ key: "execution.halted"        │
                                  │ value: "true" / "false"        │ on change:
                                  │                                │   true → stop processing
                                  │                                │   false → resume processing
```

### NATS KV Bucket

| Property | Value |
|----------|-------|
| Bucket | `EXECUTION_CONTROL` |
| Storage | FileStorage |
| MaxBytes | 1 MB |
| Key | `execution.halted` |
| Values | `"true"` / `"false"` |

### Behavior

**When `execution.halted` = `"true"`:**
- Execute binary: VenueAdapterActor stops consuming from EXECUTION_EVENTS. In-flight orders are completed but no new orders are started.
- Derive binary: Execution evaluator actors stop producing intents. Upstream chain (observation → risk) continues unaffected.
- Store binary: Continues materializing — does not halt. If execute stops publishing, there's nothing new to materialize.
- Gateway binary: Continues serving queries. Kill switch does not affect read-side.

**When `execution.halted` = `"false"` (or key deleted):**
- Execute binary: VenueAdapterActor resumes consuming.
- Derive binary: Execution evaluator actors resume producing.
- Resume is logged at WARN level with timestamp.

### Activation Flow

```
1. Operator runs: configctl execution halt
     │
     ├── Writes execution.halted = "true" to EXECUTION_CONTROL KV
     ├── Writes audit log: "execution halted by operator at <timestamp>"
     │
2. Execute binary VenueAdapterActor watches EXECUTION_CONTROL
     │
     ├── Receives key change notification (NATS KV watcher)
     ├── Sets internal halted flag
     ├── Completes in-flight order (if any)
     ├── Stops consuming new intents
     ├── Logs: "execution halted — kill switch activated"
     │
3. Derive binary execution evaluator actors watch EXECUTION_CONTROL
     │
     ├── Receives key change notification
     ├── Sets internal halted flag
     ├── Stops producing new intents
     ├── Continues receiving risk assessments (drops silently, counted in stats)
     ├── Logs: "execution intent production halted — kill switch activated"
```

### Recovery

```
1. Operator runs: configctl execution resume
     │
     ├── Writes execution.halted = "false" to EXECUTION_CONTROL KV
     ├── Writes audit log: "execution resumed by operator at <timestamp>"
     │
2. Execute + derive actors detect key change → resume normal operation
```

### Kill Switch Invariants

| ID | Invariant |
|----|-----------|
| KS-1 | Kill switch affects execution actors only. Upstream chain is never halted. |
| KS-2 | Kill switch takes effect within 1 event cycle of the NATS KV watch notification. |
| KS-3 | Kill switch activation and deactivation are logged at WARN level with operator context. |
| KS-4 | Kill switch does not cause data loss. In-flight orders complete. Queued intents remain in NATS stream. |
| KS-5 | Kill switch is stateless on resume — actors resume from their consumer's last ACK position. |
| KS-6 | Kill switch state is queryable: `GET /healthz` includes `execution_halted: true/false`. |
| KS-7 | configctl MUST NOT have a kill switch bypass. Only explicit `configctl execution resume` deactivates it. |

## 9. Trace Persistence Strategy

### Problem (from S74 HB-4)

KV projections store only domain model fields. `correlation_id` and `causation_id` flow through the system (logs + NATS headers) but are not queryable from the HTTP surface.

### Decision: Embed in KV Payload

**Option 1 (CHOSEN)**: Embed `correlation_id` and `causation_id` directly in the KV projection payload.

**Rationale:**
- Simplest implementation — no new buckets or streams
- Consistent with the existing KV payload pattern
- Queryable from the existing HTTP endpoint
- No operational complexity increase

### Implementation Design

**Domain model extension:**

```go
type ExecutionIntent struct {
    // existing fields...
    Type       string            `json:"type"`
    Source     string            `json:"source"`
    Symbol     string            `json:"symbol"`
    Timeframe  int               `json:"timeframe"`
    Side       Side              `json:"side"`
    Quantity   string            `json:"quantity"`
    Status     Status            `json:"status"`
    Risk       RiskInput         `json:"risk"`
    Parameters map[string]string `json:"parameters"`
    Metadata   map[string]string `json:"metadata"`
    Final      bool              `json:"final"`
    Timestamp  time.Time         `json:"timestamp"`

    // Trace context (Phase 2 addition)
    CorrelationID string `json:"correlation_id,omitempty"`
    CausationID   string `json:"causation_id,omitempty"`
}
```

**Projection actor change:**
- `ExecutionProjectionActor.onExecution()` already has access to `msg.Event.Metadata.CorrelationID` and `msg.Event.Metadata.CausationID` (see line 164-165 of current code).
- Before storing to KV, copy trace IDs from event metadata into the intent struct.

**Query response change:**
- `GET /execution/paper_order/latest` response includes `correlation_id` and `causation_id`.
- Enables: "show me the chain that produced this order" from a single HTTP query.

### Trace Persistence for Fills

`ExecutionFill` already includes `CorrelationID` and `CausationID` in its design (Section 4). Fill projections will persist trace context from the start.

### What This Does NOT Solve

- Full causal chain query (observation → ... → execution) still requires multiple API calls or JetStream replay.
- Automated trace verification test (S68 prerequisite A-2) — that requires a separate test, not persistence.
- Cross-domain trace correlation — future concern for a dedicated audit service.

### Alternatives Rejected

| Alternative | Rejection Rationale |
|-------------|---------------------|
| Separate audit KV bucket | Extra operational complexity for first step. Consider for full audit trail. |
| Dedicated audit stream | Over-engineered for latest-only projection. Consider when history projection is built. |
| JetStream replay only (status quo) | Fails the "queryable from HTTP" requirement. |

## 10. Execute Binary Architecture

### Structure

```
cmd/execute/
  main.go          — Entry point, config loading, binary startup
  run.go           — Wire VenueAdapterActor, consumer, fill publisher

internal/actors/scopes/execute/
  venue_adapter_actor.go       — Consumes execution intents, places orders
  fill_publisher_actor.go      — Publishes fill events to EXECUTION_FILL_EVENTS
  execute_supervisor.go        — Supervises execute actors
  messages.go                  — Local actor messages

internal/adapters/nats/
  execution_fill_publisher.go  — NATS adapter for fill event publishing
  fill_kv_store.go             — KV adapter for fill projection
  fill_registry.go             — Event/control specs for fill stream
```

### VenueAdapterActor Lifecycle

```
Started:
  1. Create NATS consumer for EXECUTION_EVENTS (filter: execution.events.venue_market_order.>)
  2. Start NATS KV watcher for EXECUTION_CONTROL.execution.halted
  3. Initialize VenuePort (injected)
  4. Begin consuming

Message loop:
  1. Check kill switch flag — if halted, NAK and return
  2. Decode execution intent
  3. Staleness check on intent timestamp
  4. Idempotency check: has this intent already been submitted? (check by dedup key)
  5. Call VenuePort.SubmitOrder()
  6. On success: publish fill event, ACK
  7. On transient failure: NAK for redelivery
  8. On permanent failure: publish rejected event, ACK

Stopped:
  1. Complete in-flight order (grace period: 30 seconds)
  2. Close NATS connections
  3. Log final stats
```

## 11. Store Extensions for Venue Integration

### New Consumers

| Durable | Filter | Purpose |
|---------|--------|---------|
| `store-execution-fill` | `execution.fill.>` | Materialize fill events |

### New KV Buckets

| Bucket | Key Format | Purpose |
|--------|-----------|---------|
| `EXECUTION_FILL_LATEST` | `{venue_order_id}` | Latest fill per order |

### New Projection Actor

`FillProjectionActor` in store — follows same 3-gate pattern:
1. Final gate (fill events are always final)
2. Validation gate
3. Monotonicity guard (by venue_timestamp)

### Intent Projection Extension

`ExecutionProjectionActor` gains an additional responsibility: when a fill event arrives referencing an intent, the intent's status is updated in KV. This maintains the intent projection as the "current state of the order" view.

**Two options for status update:**
1. **Option A**: Execute binary publishes status-change events to EXECUTION_EVENTS. Store projection updates intent status on receipt.
2. **Option B**: Store projection reads fill events and infers intent status.

**Decision: Option A** — explicit status events. The execute binary knows the status; store materializes it. No inference.

## 13. Implementation Sequence

This design is implemented across S76–S80:

| Stage | Scope | Deliverables |
|-------|-------|-------------|
| **S76** | Failure recovery hardening | Publish retry utility, projection NAK pattern, stats extension |
| **S77** | Execution lifecycle + fill model | Status enum extension, transition validation, fill entity, trace persistence |
| **S78** | Kill switch + trace persistence | EXECUTION_CONTROL bucket, configctl halt/resume, trace in KV |
| **S79** | Derive actor tests + operational validation | Actor-level routing tests, live smoke-multi with execution |
| **S80** | First guarded venue adapter | Execute binary, PaperVenueAdapter, fill projection, status events |

**Critical path**: S76 → S77 → S78 → S79 → S80

No stage may skip its predecessor. Each stage has a readiness gate.
